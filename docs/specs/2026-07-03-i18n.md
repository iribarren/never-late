# Feature 08 (extra) — Internacionalización (i18n) / Localization

- **Status:** Draft — awaiting user approval
- **Date:** 2026-07-03
- **Feature branch (once approved):** `feature/i18n`
- **Tutorial lesson (deliverable):** `tutorial/08-i18n.md` (Spanish)

## Overview

Never Late Again currently ships a single, hardcoded-in-Spanish resource file
(`res/values/strings.xml`) and has a handful of user-visible strings and
locale-sensitive formatting (dates, durations, counts) scattered across
Composables, the home-screen widget and the lock-screen notification.

This feature makes the app **properly localizable**: every user-facing string
lives in string resources, the app ships **two language variants** (Spanish as
the base locale and English as the added variant), and all
numbers/dates/durations/plurals are formatted according to the **device
locale** using the Android/Java locale-aware APIs. When the user changes their
device language, the whole app — screens, widget and notification — follows.

Because this is a teaching project, the feature is scoped to introduce a
coherent, self-contained set of Android/Kotlin i18n concepts (configuration
-qualified resources, `stringResource`/`pluralStringResource`, `Locale`,
`NumberFormat`, `DateTimeFormatter`) that the Spanish lesson `tutorial/08-i18n.md`
will explain end to end.

### Base-locale decision

- The **base locale is Spanish**: `res/values/strings.xml` stays Spanish and
  remains the fallback for any device whose language has no matching qualifier.
- The **added variant is English**: a new `res/values-en/strings.xml` provides
  English translations, used automatically on English-language devices.

Rationale: the app is already authored in Spanish and the current default
resources are Spanish, so keeping Spanish as the fallback is the lowest-risk
choice and avoids re-translating the existing baseline. (Reversing this later —
English base + `values-es` — is a mechanical change if ever desired.)

## Goals

- **Zero hardcoded user-facing text.** Every string shown to the user (screens,
  widget, notification, content descriptions, error messages) comes from a
  string resource. Compose-preview/sample data is exempt (see Out of Scope).
- **Two complete, parallel translations.** `values/` (es) and `values-en/`
  contain the same set of `name`s, with no missing keys in either.
- **Locale-correct formatting.** Numbers, dates, times, durations and counts
  render using the device locale's conventions (decimal/grouping separators,
  date order, month/day names), not a hardcoded pattern or `Locale.getDefault()`
  misuse.
- **Correct plurals.** Count-dependent text (e.g. "N tareas pendientes") uses
  Android plural resources, not manual string concatenation or a fixed suffix.
- **All three surfaces follow the device language.** Changing the device
  language and reopening the app updates screens, the widget and the
  notification.
- **A Spanish tutorial lesson** (`tutorial/08-i18n.md`) that teaches the new
  concepts against the real code.

## New concepts taught (tutorial)

The lesson `tutorial/08-i18n.md` (Spanish) must cover, building on lessons 01–07:

- **Configuration-qualified resources**: how Android resource qualifiers work,
  why `values-en/` is selected on English devices, and how the base `values/`
  acts as the fallback. Relationship to feature 07's theme qualifiers is a good
  callback.
- **`stringResource` / `pluralStringResource`** in Compose, and
  `Resources.getString` / `getQuantityString` for non-Compose surfaces (widget,
  notification, service).
- **Placeholders and format args** (`%1$s`, `%1$d`, positional args) and **why
  string concatenation is an i18n bug** (word order differs across languages).
- **`Locale`**: what the device locale is, how to obtain it correctly for the
  current configuration, and the pitfalls of `Locale.getDefault()`.
- **Locale-aware `NumberFormat`** for numbers/counts and **`DateTimeFormatter`**
  (localized styles such as `ofLocalizedDateTime`) for dates/times, versus the
  fixed `dd/MM/aaaa HH:mm` pattern used today.
- **Plurals** via `<plurals>` resources and `pluralStringResource` /
  `getQuantityString`, including the difference between the `one`/`other`
  categories in es and en.

## User Stories

### US-1 — Read the app in my language

> As an English-speaking user, I want the whole app to appear in English when my
> device language is English, so that I can understand and use it.

**Acceptance criteria**
- With device language set to English, every screen (Onboarding, Home, Tasks
  list, Task create/edit, Articles list, Article detail, Settings) shows English
  text, including titles, buttons, labels, empty states, error messages and
  content descriptions.
- With device language set to Spanish (or any non-English language), the same
  surfaces show Spanish text (the base fallback).
- No screen shows a raw resource name (e.g. `home_greeting`) or a mixed-language
  string in either locale.

### US-2 — Widget and notification follow my language

> As a user, I want the home-screen widget and the lock-screen notification to
> be in my device language too, so that the app feels consistent everywhere.

**Acceptance criteria**
- The pending-tasks widget (title, empty state, description) is localized in
  both es and en.
- The lock-screen notification (channel name/description, title, collapsed line,
  expanded rows, redacted public summary) is localized in both es and en.
- Changing the device language and letting the widget/notification refresh shows
  the new language (a manual widget refresh / app relaunch is acceptable; live
  hot-swap without refresh is not required).

### US-3 — Correctly formatted dates and times

> As a user, I want deadlines and times to be shown in my locale's format, so
> that dates read naturally to me.

**Acceptance criteria**
- Deadline display (`tasks_deadline_label`) uses a **locale-aware** date/time
  format (e.g. `DateTimeFormatter.ofLocalizedDateTime(...)` bound to the device
  locale), not a hardcoded `dd/MM/aaaa HH:mm` pattern.
- The same instant renders with locale-appropriate ordering and separators in es
  vs en (e.g. day/month order and month names differ appropriately).
- Deadline **input parsing** in Task create/edit continues to work; if the input
  field keeps an explicit typed format, the expected pattern shown to the user
  (`task_edit_deadline_label`) is a localized string and parsing remains
  deterministic (see Risks).

### US-4 — Correctly formatted numbers and durations

> As a user, I want durations and numeric counts to be formatted for my locale,
> so that numbers look right.

**Acceptance criteria**
- Numeric counts intended for display go through locale-aware `NumberFormat`
  (integer formatting) rather than default `toString()` where a separator could
  matter.
- The duration label (`tasks_duration_label`, currently built as e.g. "1 h 30
  min") is assembled from localized string resources with format args, not from
  concatenated hardcoded unit abbreviations.

### US-5 — Correct pluralization

> As a user, I want counts like "1 tarea pendiente" vs "3 tareas pendientes" to
> be grammatically correct, so that the text never reads awkwardly.

**Acceptance criteria**
- Any count-dependent user-facing text uses a `<plurals>` resource with `one`
  and `other` categories, resolved via `pluralStringResource` (Compose) or
  `getQuantityString` (widget/notification).
- At minimum the notification public summary ("N tareas pendientes …") is
  converted from the current fixed `%1$d tareas pendientes` string to a plural
  resource, correct for both es and en.
- No user-facing count text is produced by manual string concatenation or a
  hardcoded singular/plural suffix.

### US-6 — No missing translations

> As a developer/maintainer, I want the two locales to stay in sync, so that no
> screen falls back to the wrong language for a single string.

**Acceptance criteria**
- Every `name` present in `values/strings.xml` has a counterpart in
  `values-en/strings.xml` (and vice versa); there are no orphan keys.
- Every plurals resource defines the categories required for both es and en.
- A lint/inspection pass (Android Studio "Translation" inspection or `lint`)
  reports no missing-translation or extra-translation warnings for the two
  locales.

## Technical Approach

High-level, local-only, resource-driven. No new module.

1. **Audit & extraction.** Sweep `app/src/main/java/com/neverlate/` for
   user-facing literals in Composables, the widget (`ui/widget`), and the
   notification (`ui/notification`, `TasksNotificationService`,
   `TasksNotificationHelper`). Confirmed audit targets so far: hardcoded
   unit/format assembly in `data/tasks/TaskTiming.kt` (duration label) and the
   fixed deadline pattern in the task edit/display path. Extract any remaining
   literals into `res/values/strings.xml`. (Sample/preview data such as
   "Preparar la presentación" in `@Preview` Composables and the seeded article
   content are **not** in scope — see Out of Scope.)
2. **English variant.** Create `res/values-en/strings.xml` mirroring every key,
   including `<plurals>`.
3. **Plurals.** Introduce `<plurals>` for count-dependent text (notification
   public summary at minimum) and replace concatenation-based counts.
4. **Locale-aware formatting.** Replace fixed date/duration patterns with
   `DateTimeFormatter` (localized styles) and `NumberFormat`, bound to the
   current configuration's `Locale`. Centralize formatting helpers (e.g. in a
   `domain`/util location alongside `TaskTiming`) so screens, widget and
   notification share one locale-correct implementation. Keep any **machine**
   parse/format (e.g. persisted values, deadline input parsing) explicitly
   locale-independent and separate from **display** formatting.
5. **Verification.** Manually switch device/emulator language es↔en and confirm
   all three surfaces update; run the Android Studio Translation inspection.
6. **Tutorial.** Write `tutorial/08-i18n.md` (Spanish) walking through the code.

## Out of Scope

- **In-app language picker that overrides the device locale.** The app follows
  the device language only. (Per-app language preferences via
  `AppCompatDelegate.setApplicationLocales` / `locales_config.xml` may be
  mentioned in the tutorial as an **optional stretch**, but are not built here.)
- **RTL-specific work beyond what Compose gives for free.** No dedicated
  right-to-left layout, mirroring or bidi testing; only the two LTR languages
  (es, en) are targeted.
- **More than one additional language.** Only English is added. No third locale,
  no region-specific variants (e.g. `values-es-rMX`, `values-en-rGB`).
- **Translating dynamic/user-authored content.** Task titles the user types and
  the seeded article bodies are not machine-translated or duplicated per locale.
- **Localizing Compose `@Preview`/sample data and test fixtures.** These may stay
  hardcoded; they are not user-facing.
- **Translation tooling / TMS integration, string freeze process, or
  pseudolocalization** beyond a one-time manual/lint check.
- **Currency, measurement-system, or timezone conversion** features (only
  number/date/plural formatting for the existing displayed values).

## Dependencies

- No new libraries expected. Uses the Android resource system, Compose
  `stringResource`/`pluralStringResource` (already available), and
  `java.text.NumberFormat` / `java.time.format.DateTimeFormatter` (JDK, `minSdk
  = 24` — `java.time` is available; confirm whether core library desugaring is
  already enabled and, if not, whether any used `java.time` API needs it).
- Relies on the existing surfaces being resource-ready: feature 06 notification
  strings and feature 05 widget strings already exist in `values/strings.xml`
  and only need an `en` counterpart plus plural/format adjustments.
- No backend, no DataStore/Room schema change. (This feature is orthogonal to the
  feature 07 theme preference; both are device/config-driven.)
- Tutorial lesson `tutorial/08-i18n.md` must be authored before commit
  (Documentation Update / Tutorial Methodology are binding).

## Risks

- **Deadline input parsing vs. localized display.** Today the edit screen both
  shows and (presumably) parses `dd/MM/aaaa HH:mm`. Making *display* locale-aware
  while keeping *input parsing* deterministic can diverge: a localized hint that
  doesn't match the parser will confuse users. Mitigation: keep input parsing on
  a fixed, documented pattern (or a picker) and localize only the hint text and
  the read-only display; call this out in the lesson.
- **`Locale.getDefault()` staleness in long-lived surfaces.** The foreground
  service / widget worker may capture a locale at process start; after a language
  change the correct locale must be read from the current `Configuration`/context
  at render time. Mitigation: resolve locale per render, not once at
  construction.
- **`java.time` + `minSdk = 24`.** Some `DateTimeFormatter`/`java.time` niceties
  require core library desugaring below API 26. Risk of a build/runtime surprise
  if desugaring isn't enabled. Mitigation: verify the desugaring flag early;
  prefer APIs safe on API 24 or enable desugaring in the version catalog/gradle.
- **Translation drift / missing keys.** Adding `values-en` doubles the surface to
  maintain; a missed key silently falls back to Spanish on an English device.
  Mitigation: rely on the Translation lint inspection as an acceptance gate
  (US-6).
- **Plural categories.** es and en both use `one`/`other`, so risk is low, but
  the tutorial should note that other languages need more categories — important
  since a future third language would break naive assumptions.
- **Scope of the audit.** "All UI text" is only as good as the sweep; some string
  may hide in a less obvious place (e.g. accessibility descriptions,
  toasts/snackbars). Mitigation: grep-based audit plus a manual click-through of
  every screen in both locales.
