# Feature 13d (extra) — Migrar la inyección de dependencias a Hilt

- **Status:** Draft — awaiting approval
- **Date:** 2026-07-12
- **Type:** Behaviour-preserving refactor + tutorial lesson (Dependency Injection)
- **Branch (suggested):** `feature/hilt-di`
- **Backlog slot:** `docs/conceptos-pendientes.md` §5 (Inyección de dependencias)
- **Lesson:** `tutorial/13d-hilt-di.md` (Spanish, numbered **13d**, between 13 and 14)

## Overview

Since **feature 02** the app has wired its dependencies **by hand**: a manual
`ViewModelProvider.Factory` (`ui/navigation/AppViewModelFactory`) builds every screen's
`ViewModel`, singletons are created ad-hoc, and the `TaskRepository` decorator chain is composed
inline inside `MainActivity.onCreate`. That was the right call while the graph was small and the
lesson plan hadn't reached DI yet.

With the backend + sync + auth layers landed (features **11–13**), that manual wiring is now at its
most tedious: `MainActivity.onCreate` builds `EncryptedTokenStorage`, `AuthRepositoryImpl`,
`TasksNetwork`, `SyncEngine`, a three-deep repository decorator chain, and threads a bag of
repositories down through `AppNavHost` into `AppViewModelFactory`. This is the **perfect, motivated
teaching moment** to introduce a real DI container — the reader has just *felt* the pain the tool
solves.

This feature is therefore an **"antes/después" refactor, not a new product feature**. It migrates
the app's object construction and wiring to **Hilt** (Dagger) while preserving **100% of observable
behaviour**: same seams, same decorator composition (and order), same singletons-per-process, same
navigation, same UI. Nothing the user can see or do changes.

> **Scope discipline:** this is a wiring swap. The seams (`TaskRepository`, the sync/reminder
> decorators, `AuthRepository`, `ArticleRepository`, `UserPreferencesRepository`) do **not** change
> their interfaces or behaviour. "Extend, don't duplicate" applies to wiring exactly as it applies
> to logic elsewhere in the tutorial.

## Goals

- Introduce **Hilt** as the project's dependency-injection container, teaching DI from first
  principles.
- **Retire** the manual DI: delete `AppViewModelFactory` and remove all object *construction/wiring*
  from `MainActivity.onCreate`.
- Preserve behaviour **provably**: the decorator chain order, the guest-mode adoption trigger, all
  seams, and all imperative startup side-effects are unchanged.
- Keep the whole existing test battery (unit + instrumented) green with **no** test-logic changes
  forced by the refactor (test wiring may adapt, e.g. `HiltAndroidRule`, but assertions/behaviour
  under test do not).
- Ship the Spanish lesson `tutorial/13d-hilt-di.md`.

## What this feature is (and is not)

**It is:** replacing *how* objects are constructed and handed to `ViewModel`s and `MainActivity`
with a Hilt object graph.

**It is not:** a change to *what* those objects are, how they behave, what they're composed of, or
in what order. No new screens, no new strings, no new colours, no backend/contract change, no
DB-version bump, no permission change, no behavioural change of any kind.

## Concepts the lesson must teach (Spanish `tutorial/13d-hilt-di.md`)

Following the Tutorial Methodology (each feature teaches progressively harder, new concepts):

1. **Por qué DI** — the concrete cost of manual DI in *this* codebase (point at
   `MainActivity.onCreate` + `AppViewModelFactory` before/after), and what a container buys you:
   no hand-threading, automatic transitive resolution, single source of scope truth.
2. **Hilt básico** — `@HiltAndroidApp` (the generated `Application` component), constructor
   `@Inject`, `@Module`, `@Provides` vs `@Binds` (and *when* each: `@Binds` for interface→impl,
   `@Provides` for constructed/third-party objects), `@Singleton`, and Hilt
   **components/scopes** (`SingletonComponent`).
3. **Hilt + Compose + Navigation** — `@HiltViewModel`, `hiltViewModel()`, and how a Hilt
   `ViewModel` gets its lifecycle/scope from the nav back-stack entry; how a **navigation argument**
   (e.g. `articleId`, `taskId`) reaches a `ViewModel` via `SavedStateHandle` instead of a factory
   parameter.
4. **Proveer una cadena de decoradores con Hilt** — the crux: providing an ordered decorator chain
   through Hilt using **qualifiers**, so each layer receives exactly the right inner instance and
   the composition order is unambiguous and unchanged.

## Critical constraint — decorator chain order (MUST preserve exactly)

The `TaskRepository` seam is a **three-layer decorator chain** currently composed inline in
`MainActivity.onCreate`. Hilt MUST reproduce it **outermost → innermost, in this exact order**:

```
TaskSurfacesRefreshingRepository   (feature 05/06 — refresh widget + lock-screen summary)
  └─ ReminderSchedulingRepository  (feature 09 — schedule/cancel each task's alarm)
       └─ OutboxTaskRepository     (feature 11 — stamp sync metadata + enqueue outbox row)
            └─ RoomTaskRepository  (real, Room-backed source of truth)
```

Because Hilt sees four things that all satisfy the `TaskRepository` type, this **requires
qualifiers** (e.g. `@RoomRepo`, `@OutboxRepo`, `@ReminderRepo`, and the un-qualified/`@Singleton`
outermost one that the rest of the app injects) so each `@Provides`/`@Binds` receives precisely the
correct inner instance and there is no ambiguity. The order is a behavioural contract — every write
must still refresh surfaces, sync reminders, *and* queue for the backend, in that sequence.

Additional wiring that MUST be preserved through the migration:

- **Guest-mode adoption trigger (feature 13):** `authRepositoryImpl.onAuthenticated =
  { taskRepository.refreshFromServer() }`. This late-binding hook (task repo must exist before it
  can be wired) must survive — provided/wired via Hilt without changing the mechanism. It is
  *redundant insurance* alongside `AppNavHost`'s `LaunchedEffect`, not a replacement for it.
- **`AppNavHost` `Guest`/`LoggedIn` separate `when` arms:** these MUST remain separate arms so a
  `Guest → LoggedIn` transition recomposes a fresh `LaunchedEffect { refreshFromServer() }` —
  **do not merge them**. Migrating to `hiltViewModel()` must not collapse this.
- **`SyncEngine` / `TasksNetwork` / `AuthInterceptor` / `TokenAuthenticator` graph:** the token
  storage feeds the tasks network client's auth interceptor, and `onUnauthorized` routes back to
  `AuthRepositoryImpl.notifyUnauthorized`. This dependency shape is reproduced by Hilt providers
  unchanged.

**Imperative startup side-effects stay in `MainActivity.onCreate`.** Only object
*construction/wiring* moves to Hilt. These remain, called imperatively in `onCreate`:

- `ReminderNotificationHelper.ensureChannel(...)`
- `TaskSurfacesRefreshWorker.enqueuePeriodic(...)`
- `SyncWorker.enqueuePeriodic(...)`
- `TasksNotificationService.refresh(...)`
- reading `openTasksOnStart` from the launch `Intent`
- the theme-preference collection inside `setContent` that drives `NeverLateTheme`

(Dependencies these side-effects need — e.g. an application `Context`, or repositories — may be
`@Inject`ed into `MainActivity` rather than constructed there.)

## Technical Approach

App-only (the `app/` Android module). No change to `backend/`, `docs/api/contract.md`, DB version,
manifest permissions, or any seam interface.

- **Gradle plugins:** apply `com.google.dagger.hilt.android` in the root `build.gradle.kts`
  (`apply false`) and the app `build.gradle.kts`. Versions live in the version catalog
  (`gradle/libs.versions.toml`) — never hardcoded.
- **Application class:** new `NeverLateApplication` (or similar) annotated `@HiltAndroidApp`,
  registered in `AndroidManifest.xml` via `android:name`.
- **Entry point:** `@AndroidEntryPoint` on `MainActivity`.
- **Modules** under a new `di/` package (`SingletonComponent`-scoped, `@Singleton` where the current
  code uses process-singletons):
  - **Database module** — provides `NeverLateDatabase` (via `getInstance(context)`) and its DAOs
    (`taskDao`, article DAOs, remote-keys DAO, etc.).
  - **Network module** — provides OkHttp/Retrofit and the API interfaces: `ArticlesApi`,
    `AuthNetwork`/auth API, `TasksApi`/`TasksNetwork` (including the `AuthInterceptor` +
    `TokenAuthenticator` wiring and the `onUnauthorized` callback), and `SyncEngine`.
  - **Storage module** — provides `TokenStorage` bound to `EncryptedTokenStorage`, and
    `UserPreferencesRepository` bound to `DataStoreUserPreferencesRepository`.
  - **Repository module** — provides `AuthRepository` (bound to `AuthRepositoryImpl`),
    `ArticleRepository` (bound to `CachingArticleRepository`), the `ReminderScheduler`, and the
    **qualified `TaskRepository` decorator chain** in the exact order above.
- **ViewModels:** annotate all nine with `@HiltViewModel` + `@Inject constructor`; nav-arg
  ViewModels read arguments from `SavedStateHandle`. Compose call sites switch to `hiltViewModel()`.
- **`@Provides` vs `@Binds`:** interface→single-impl bindings (`TokenStorage`,
  `UserPreferencesRepository`, `AuthRepository`, `ArticleRepository`, inner `RoomTaskRepository`)
  favour `@Binds`; constructed/third-party/decorator-composition objects use `@Provides`.

### Implementation decision to make (flag in the PR)

- **KSP vs kapt for Hilt's compiler.** The project already uses **KSP** for Room, so KSP is
  preferred for a single annotation-processing pipeline. Hilt's Gradle plugin supports KSP in recent
  versions, but historically Hilt's compiler required **kapt**. The implementer must pick the
  version that supports KSP; if the pinned Hilt version cannot, fall back to kapt (adding the kapt
  plugin) and note the reason. Record the outcome in the lesson and the PR.

- **No new screens, strings, colours, layouts, or any UI change.**

## Risks

- **Decorator order regression (highest risk).** Getting Hilt qualifiers subtly wrong could reorder
  or drop a decorator layer, silently breaking widget refresh, reminders, or sync. *Mitigation:*
  explicit qualifiers, a clear provider that reads top-to-bottom like the current inline code, and
  the existing behavioural tests as the safety net; call this out in the lesson.
- **Guest-mode trigger lost.** Collapsing the `Guest`/`LoggedIn` nav arms or dropping the
  `onAuthenticated` hook while refactoring would silently break sign-in adoption. *Mitigation:*
  explicit acceptance criteria; do not touch `AppNavHost`'s arm structure.
- **Nav-arg ViewModels via `SavedStateHandle`.** Moving `articleId`/`taskId` from factory params to
  `SavedStateHandle` can change null/absent semantics if done carelessly (esp. `TaskEditViewModel`'s
  "absent taskId is valid"). *Mitigation:* preserve the exact current semantics; cover with the
  existing tests.
- **KSP/kapt friction.** If the chosen Hilt version doesn't support KSP, adding kapt alongside KSP
  can slow builds and needs correct plugin ordering. *Mitigation:* pick a KSP-capable Hilt version;
  document the fallback if not.
- **Test-harness churn.** Instrumented tests may need `HiltAndroidRule`/a test application. *Risk* is
  that this is mistaken for a behavioural change. *Mitigation:* only wiring in tests may change,
  never the asserted behaviour.
