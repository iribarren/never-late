## Overview

Today the app still ships the Android Studio template's default Material 3 identity: the
purple placeholder palette in `ui/theme/Color.kt` (`Purple80`/`Purple40`/`PurpleGrey*`/`Pink*`),
a near-default `Typography` in `ui/theme/Type.kt` (only `bodyLarge` is customised), and
`NeverLateTheme` with `dynamicColor` **hard-defaulted to `true`**. On Android 12+ that means the
device wallpaper â€” not the product â€” decides the app's colors, so "Never Late Again" has no
recognizable visual identity of its own.

This feature gives the app a **brand-owned identity** in three moves:

1. Replace the template palette with a **brand Material 3 color scheme** (full color roles, light
   and dark), generated with **Material Theme Builder** from the direction in
   `docs/mockups/rediseno-ux-ui.html` (brand primary `#3B5BDB`, brand container `#DBE4FF`, and a
   semantic urgency scale `ok #2F9E44` / `soon #F08C00` / `late #E03131`). The HTML is directional
   guidance only â€” not code to copy, and not a hand-picked color list.
2. Define a **brand typographic scale** in `Type.kt` (`displayLargeâ€¦labelSmall`), optionally backed
   by a **custom font** loaded from `res/font/` via `FontFamily`.
3. Add a **"Material You (dynamic color)" toggle** in Settings so the user chooses between the brand
   palette ("app color") and the wallpaper-driven system palette ("system color"). This is one more
   preference on the **existing feature-07 seam** (`user_prefs` DataStore â†’ `SettingsViewModel` â†’
   `NeverLateTheme`), not a new theming system.

As a teaching project, the feature must introduce genuinely new concepts and ship its Spanish
lesson (see Tutorial Methodology in `CLAUDE.md`).

## Goals

- The app looks like *itself* on every device and Android version by default: the brand palette and
  type scale are what a first-run user sees, independent of their wallpaper.
- Users who prefer Material You can opt back into wallpaper-driven color with one setting, and that
  choice survives app restarts.
- The existing preference infrastructure is **extended, not duplicated** â€” the new toggle rides the
  same `user_prefs` DataStore, `UserPreferences`, `SettingsViewModel`, and `NeverLateTheme` seam as
  `ThemeMode`.
- The lesson teaches Material 3 color *roles*, the typographic scale + custom fonts, and the
  `dynamicColor` (brand vs system) trade-off.

## User Stories

### US-1 â€” Brand color scheme (light & dark)

> As a user, I want the app to use its own brand colors instead of the template purple, so that
> "Never Late Again" feels like a finished, recognizable product.

**Acceptance criteria**
- `ui/theme/Color.kt` no longer defines the template placeholders (`Purple80`, `PurpleGrey80`,
  `Pink80`, `Purple40`, `PurpleGrey40`, `Pink40`); it holds a **full set of Material 3 role tokens**
  for both schemes, generated by Material Theme Builder (primary, onPrimary, primaryContainer,
  onPrimaryContainer, secondary, tertiary, background, surface, surfaceVariant, outline, error, etc.).
- `ui/theme/Theme.kt`'s `LightColorScheme` and `DarkColorScheme` are rebuilt from those tokens via
  `lightColorScheme(...)` / `darkColorScheme(...)`, wiring **all** generated roles (not just
  `primary`/`secondary`/`tertiary` as today).
- The brand primary is derived from the mockup seed `#3B5BDB` (not required to be pixel-identical â€”
  Material Theme Builder tonal output governs the exact hex values).
- With dynamic color **off**, the brand scheme renders in both light and dark modes with no
  remaining purple anywhere in the app.
- Colors are consumed via `MaterialTheme.colorScheme.*` role names in UI code; no screen hard-codes
  raw hex values for brand color.

### US-2 â€” Brand typography

> As a user, I want the app's text to use a consistent brand type scale, so that the UI reads as one
> designed system rather than raw defaults.

**Acceptance criteria**
- `ui/theme/Type.kt` defines a **complete** `Typography` where the brand scale is applied across the
  Material 3 roles used by the app (at minimum `displayLarge`, `headlineMedium`/`headlineSmall`,
  `titleLarge`, `bodyLarge`/`bodyMedium`, `labelLarge`/`labelSmall`), replacing the current
  single-`bodyLarge` near-default.
- If a custom font is introduced, the font files live in `res/font/` and are loaded through a
  `FontFamily` (with weights), and that `FontFamily` is referenced by the type scale; if no custom
  font is used, the choice to stay on a system family is deliberate and noted in the lesson.
- Existing screens that read `MaterialTheme.typography.*` (e.g. `SettingsScreen`'s
  `bodyLarge`/`bodyMedium`) pick up the new scale without per-screen changes.

### US-3 â€” Choose brand color vs Material You

> As a user on Android 12+, I want to choose between the app's own colors and my system (wallpaper)
> colors, so that the brand isn't silently overridden by Material You.

**Acceptance criteria**
- Settings gains a control (in the existing theme section card) to turn **dynamic color / Material
  You on or off**, labeled for users (e.g. "App color" vs "System color (Material You)"), with all
  strings in both `res/values/strings.xml` (Spanish base) and `res/values-en/strings.xml`.
- The default for a fresh install is **dynamic color OFF** (brand wins by default), so the app's
  identity is what a new user sees.
- On devices below Android 12 the toggle is either hidden or clearly shown as unavailable, since
  `dynamicColor` has no effect there (`Build.VERSION.SDK_INT < S`) â€” the app still uses the brand
  scheme.
- Toggling the setting recomposes the whole app's colors live (the value is read at the top of the
  composition in `MainActivity`, same place `themeMode`/`darkTheme` are read today).

### US-4 â€” Preference persists and reuses the existing seam

> As a returning user, I want my color choice remembered across restarts, so that I don't have to
> reset it every launch.

**Acceptance criteria**
- A new `dynamicColor` (or equivalently named) field is added to `UserPreferences` with a boolean
  `Keys` entry in the **same `user_prefs` DataStore** â€” no second DataStore is created.
- The repository reads it tolerantly (missing key â†’ the default) exactly like `themeMode`,
  `remindersEnabled`, etc., and exposes a `saveDynamicColor(enabled: Boolean)` on
  `UserPreferencesRepository` + `DataStoreUserPreferencesRepository`.
- `SettingsUiState` carries the value and `SettingsViewModel` gains an
  `onDynamicColorChanged(enabled)` that writes through the repository; the UI updates reactively by
  observing the preferences flow (same pattern as `onThemeModeSelected`).
- After changing the setting and fully restarting the app, the previously chosen value is still in
  effect.

## Technical Approach

**Same seam, one more preference â€” do not build a parallel system.**

- **Palette (`Color.kt` + `Theme.kt`):** Generate the scheme in Material Theme Builder from the
  mockup direction, export the Kotlin color tokens into `Color.kt`, and rebuild
  `LightColorScheme`/`DarkColorScheme` from them. Keep `NeverLateTheme`'s existing structure: the
  `when` block still chooses `dynamic{Light,Dark}ColorScheme(context)` on Android 12+ **only when
  the new preference is on**, otherwise the brand `Light/DarkColorScheme`.
- **`dynamicColor` parameter:** `NeverLateTheme` already accepts `dynamicColor: Boolean`. The change
  is to stop defaulting it to `true` at the call site: `MainActivity` (around the existing
  `NeverLateTheme(darkTheme = darkTheme)` call, lines ~136â€“140) reads
  `userPreferences?.dynamicColor` alongside `themeMode`, falling back to the default while DataStore
  is still loading (same null-guard reasoning already used for `themeMode`), and passes it in:
  `NeverLateTheme(darkTheme = darkTheme, dynamicColor = dynamicColor) { ... }`.
- **Preference plumbing:** Extend `UserPreferences`, `DataStoreUserPreferencesRepository.Keys`, the
  `userPreferences` mapping, and add `saveDynamicColor` â€” mirroring the `themeMode`/`remindersEnabled`
  pattern already in the file.
- **Settings UI:** Add a row to the existing theme `SettingsSectionCard` in `SettingsScreen.kt`
  (a `Switch` row, consistent with the reminders on/off control), wired to
  `SettingsViewModel.onDynamicColorChanged`. Gate it on `Build.VERSION.SDK_INT >= S`.
- **Typography:** Rebuild `Typography` in `Type.kt`; if a custom font is added, drop the files under
  `res/font/` and build a `FontFamily`.
- **Urgency scale:** The mockup's `ok`/`soon`/`late` semantic colors are noted as directional. If
  they are surfaced in-app they should map onto Material roles (e.g. `error` / `tertiary`) or a
  small explicitly-documented set of extra brand tokens â€” full task-urgency theming beyond wiring
  the scheme is **out of scope** for this feature (see below).

## Out of Scope

- Any backend, API-contract, database-version, permission, or new-dependency change (this feature is
  purely client-side theming; Material Theme Builder is a design tool, not a runtime dependency).
- Re-skinning individual screens, adding new components, or a broad UI redesign beyond adopting the
  new color scheme and type scale.
- A full semantic **urgency-color system** applied to task rows/badges (the `ok`/`soon`/`late` scale
  is captured as direction here but its per-widget application is a later feature).
- Per-screen or user-editable custom themes / theme marketplace / arbitrary accent picking.
- Changing the existing `ThemeMode` (light/dark/system) behaviour â€” that seam is reused, not
  redesigned.
- Animated theme transitions.

## Risks

- **New concept vs "rehash" (teaching mandate):** the toggle rides an existing seam, so the lesson
  must center the genuinely new material (color roles, type scale, custom fonts, Material You
  trade-off) to satisfy Tutorial Methodology. Mitigate by framing the lesson around design-system
  concepts, not the plumbing.
- **Material Theme Builder fidelity:** the tool's tonal output won't match the mockup hex values
  exactly; acceptance criteria therefore anchor on *seed direction*, not exact hex.
- **Default choice (dynamic OFF):** defaulting Material You off is deliberate (brand-first) but is a
  visible product decision for existing users on Android 12+ whose app currently follows their
  wallpaper â€” call it out at approval.
- **Pre-Android-12 UX:** the toggle is inert below API 31; hiding vs disabling it is a small UX
  decision to confirm.
- **`fallbackToDestructiveMigration` is unrelated here** â€” this feature touches no Room schema, so
  no data loss risk.
